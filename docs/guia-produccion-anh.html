1. Resumen ejecutivo

El VIP (Validador del Inventario de Pozos) es una aplicación web full-stack en un solo repositorio Next.js. Hoy funciona como demostración funcional: valida Excel oficial, gestiona cuadernos por operadora, consolida inventario en panel ANH y genera UWI fiscalizado. Sin embargo, varios componentes están en modo simulación o no persistentes, inadecuados para producción institucional sin trabajo adicional.

Bloqueadores actuales para producción real
  • SQLite en Vercel se guarda en /tmp y se pierde al reiniciar la función serverless.
  • Autenticación local con usuarios y contraseñas en BD; no hay SSO/LDAP institucional.
  • Correo simulado: los envíos se escriben en data/outbox/, no salen por SMTP ANH.
  • Repositorio personal (PhDRedondo/Inventario-de-pozos), no bajo gobernanza TI ANH.
  • Credenciales demo (Anh2026!) deben eliminarse o rotarse en producción.

Objetivo de producción

Un despliegue donde:

  1. Los datos de pozos, cuadernos, usuarios y auditoría persisten en base gestionada ANH.
  2. Operadoras y funcionarios ANH acceden con identidad institucional.
  3. Al aplicar un envío, se notifica por correo real a correspondenciaanh@anh.gov.co.
  4. El código vive en GitHub de la ANH con CI/CD, revisiones y secretos controlados.
  5. La app corre en hosting aprobado (Vercel institucional u otra plataforma ANH) con dominio *.anh.gov.co.

5. Repositorio GitHub — migración a la ANH

Situación actual

  • Repositorio: github.com/PhDRedondo/Inventario-de-pozos
  • Rama principal: main
  • Sin pipelines CI formales en el repo (deploy manual o script scripts/vercel-deploy.sh)

Pasos para transferir a GitHub ANH

  1. Crear organización o usar la existente (p. ej. github.com/anh-colombia o equivalente interno).
  2. Crear repositorio inventario-pozos-vip (privado).
  3. Transferir o mirror: 
    git clone --mirror https://github.com/PhDRedondo/Inventario-de-pozos.git
cd Inventario-de-pozos.git
git remote set-url origin https://github.com/ANH-ORG/inventario-pozos-vip.git
git push --mirror
  4. Configurar branch protection en main: PR obligatorio, 1–2 revisores, sin push directo.
  5. Definir ramas: main (prod), staging, develop (opcional).
  6. Agregar CODEOWNERS para src/lib/auth*, middleware.ts, src/app/api/.
  7. Activar Dependabot o escaneo de vulnerabilidades (GitHub Advanced Security si está disponible).
  8. Documentar en README interno: contactos, runbooks, enlaces a Vercel/BD.
  9. Revocar acceso del repositorio personal una vez validada la migración.
Licencia y propiedad intelectual Formalizar que el código desarrollado para la ANH es propiedad de la entidad y que colaboradores externos pierden acceso de escritura tras la entrega.

6. Base de datos — de SQLite a gestionada ANH

No desplegar producción con SQLite en Vercel. El archivo inventario.db vive en /tmp/inventario-pozos-anh-data/ (src/lib/paths.ts). Cada cold start o redeploy puede borrar datos.

Tablas principales a migrar

TablaContenidoCriticidad
wellsInventario de pozos (40 atributos + UWI)Crítica
uploadsLotes / versiones de cargueCrítica
validation_issuesHallazgos por pozoAlta
notebooksCuadernos de inventario por operadoraCrítica
notebook_eventsTrazabilidad (timeline)Alta
usersUsuarios y rolesCrítica
audit_logAuditoría adminAlta

Trabajo de desarrollo requerido

  1. Elegir motor con TI ANH (PostgreSQL recomendado por compatibilidad y costo).
  2. Provisionar instancia en red accesible desde el hosting (VPC, firewall, IP allowlist).
  3. Introducir capa de acceso (Prisma/Drizzle/Knex) o adaptar SQL manual:
    • Reemplazar better-sqlite3 — no funciona en Vercel serverless sin filesystem persistente.
    • Convertir INTEGER PRIMARY KEY AUTOINCREMENT → SERIAL / IDENTITY.
    • Reemplazar ensureColumn ad hoc por migraciones versionadas (Flyway, Liquibase, Prisma Migrate).
    • Revisar tipos TEXT para fechas → TIMESTAMPTZ si aplica.
  4. Variables de entorno (ejemplo PostgreSQL): 
    DATABASE_URL=postgresql://vip_app:***@db-interna.anh.gov.co:5432/inventario_pozos?sslmode=require
  5. Seed controlado: en producción, cargar catálogos oficiales desde BD o API, no solo seed.json estático.
  6. Backups: snapshots diarios, retención 30–90 días, prueba de restauración trimestral.
  7. Conexiones serverless: usar pooler (PgBouncer, Neon pooler) para evitar agotar conexiones en Vercel.

Si la ANH prefiere SQL Server u Oracle

El esfuerzo es mayor: revisar todos los queries en src/lib/db.ts, notebook-db.ts, auth-db.ts, analytics-db.ts. Presupuestar 2–4 semanas adicionales de adaptación y pruebas.

Script actual de referencia: scripts/vercel-deploy.sh (genera SESSION_SECRET con OpenSSL).

Opción B: Infraestructura on-prem / nube soberana ANH

  • Contenedor Docker Node.js 20 LTS ejecutando npm run build && npm start (puerto 3000).
  • Reverse proxy Nginx/IIS con TLS terminado en el balanceador institucional.
  • Permite mantener SQLite solo si hay volumen persistente — no recomendado; preferir PostgreSQL igualmente.
  • Mayor control para auditorías MinTIC; mayor carga operativa (parches, escalado, HA).

Requisitos mínimos de runtime

  • Node.js ≥ 20
  • Memoria: ≥ 512 MB por instancia (recomendado 1 GB para cargas Excel grandes)
  • Timeout de funciones: aumentar en Vercel si cargas > 8 MB (config maxDuration en route handlers)
  • Almacenamiento temporal para parsing Excel durante la request

8. Autenticación institucional

Implementación actual

  • Login en /api/auth/login valida usuario/contraseña contra tabla users.
  • Cookie anh_session firmada con HMAC-SHA256 (SESSION_SECRET).
  • Roles: operadora, anh, admin — alcance en auth-scope.ts.
  • Usuario admin semilla: johan.redondo@anh.gov.co (contraseña ANH_ADMIN_PASSWORD o default demo).

Objetivo producción

  1. Integrar IdP institucional (Azure AD, Keycloak, LDAP) vía OIDC/SAML.
  2. Flujo recomendado: Next.js Auth.js (Auth.js v5) o middleware OIDC → sesión propia o JWT de corta vida.
  3. Mapeo de grupos AD → roles VIP:
    • GOP-Operadoras → rol operadora + atributo operadora
    • GOP-ANH-Consulta → rol anh
    • GOP-Admin → rol admin
  4. Mantener tabla users como espejo de identidades o eliminar contraseñas locales.
  5. Desactivar usuarios demo (demo, funcionario) y rotar ANH_ADMIN_PASSWORD.
  6. Habilitar MFA en el IdP para admin y funcionarios ANH.
  7. Política de sesión: expiración 8 h inactividad; cookie Secure, HttpOnly, SameSite=Lax (ya parcialmente aplicado).
Operadoras externas Si las operadoras no están en el AD ANH, definir: (a) B2B federado, (b) cuentas locales con MFA + invitación, o (c) portal extranet existente de la ANH como punto de entrada único.

9. Correo electrónico — de simulación a SMTP ANH

Comportamiento actual

Al aplicar envío (POST /api/notebooks/[id]/submit), sendSubmissionEmail() en src/lib/mail.ts escribe archivos .txt y .xlsx en outbox/. Destino simulado: correspondenciaanh@anh.gov.co.

Integración requerida

  1. Solicitar a TI ANH: servidor SMTP relay interno, credenciales de servicio, puerto, TLS.
  2. Variables de entorno sugeridas: 
    SMTP_HOST=smtp.anh.gov.co
SMTP_PORT=587
SMTP_USER=vip-notificaciones@anh.gov.co
SMTP_PASS=***
SMTP_FROM=vip-notificaciones@anh.gov.co
ANH_CORRESPONDENCE=correspondenciaanh@anh.gov.co
  3. Implementar envío real con nodemailer o API Graph (Microsoft 365).
  4. Adjuntar Excel generado para ControlDoc (ya se produce el buffer en el flujo de submit).
  5. Registrar en audit_log cada notificación enviada (message-id, timestamp).
  6. Cola de reintentos si SMTP falla (tabla email_queue o servicio de mensajería ANH).

10. Seguridad, red y cumplimiento

Checklist de seguridad

  • SESSION_SECRET ≥ 32 bytes aleatorios; rotación anual documentada.
  • Eliminar contraseñas por defecto y usuarios demo en producción.
  • HTTPS obligatorio; HSTS en dominio *.anh.gov.co.
  • WAF / protección DDoS según estándar ANH.
  • Rate limiting en /api/auth/login y cargas de archivo.
  • Límite de tamaño de upload Excel (p. ej. 15 MB) — validar en API route.
  • Escaneo de dependencias (npm audit, Snyk, Dependabot).
  • Logs sin datos personales sensibles; retención según política ANH.
  • Prueba de penetración antes de go-live (mínimo aplicación web OWASP Top 10).
  • Plan de respuesta a incidentes y contacto CSIRT ANH.
  • Cumplimiento Ley 1581 (datos personales en actor_email, nombres operadores).

Acceso de red

  • BD: solo desde subnets del hosting VIP (no exposición pública sin VPN).
  • Panel admin /admin/usuarios: restringir por IP institucional si es posible.
  • Operadoras: acceso vía internet con autenticación fuerte o VPN extranet ANH.

11. CI/CD — pipeline recomendado

Push/PR → GitHub ANH │ ├─ Job: lint + typecheck + build (npm run lint && npm run build) ├─ Job: test UWI (npm run test:uwi) — opcional en PR ├─ Job: escaneo seguridad dependencias │ ├─ PR merge a staging → Deploy automático Vercel Staging + smoke tests │ └─ Tag v* o merge a main (aprobación manual) → Deploy Production Vercel

GitHub Actions (plantilla conceptual)

name: CI
on: [push, pull_request]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: "20" }
      - run: npm ci
      - run: npm run lint
      - run: npm run build
        env:
          SESSION_SECRET: ci-dummy-secret-for-build-only

Secretos de producción nunca en el repositorio; usar GitHub Environments + Vercel integración.

13. Puesta en producción (go-live)

Pre-go-live (T-7 días)

  • BD producción provisionada y migraciones aplicadas.
  • Variables de entorno producción verificadas (checklist anexo).
  • SSO probado con usuarios reales de cada rol.
  • DNS vip.anh.gov.co apuntando al hosting.
  • Backup inicial y procedimiento de rollback documentado.
  • Usuarios demo deshabilitados.
  • Comunicado a operadoras (fecha, URL, soporte).

Día D

  1. Ventana de mantenimiento comunicada (si aplica).
  2. Deploy tag v1.0.0 a producción.
  3. Smoke test: login SSO, listar panel, cargar cuaderno prueba, exportar informe.
  4. Monitoreo activo primeras 48 h.
  5. Mesa de ayuda GOP + TI con canal único (correo / Teams).

Rollback

  • Vercel: promover deployment anterior estable desde dashboard.
  • BD: restaurar snapshot pre-deploy si migración falló.
  • Comunicar degradación a operadoras si se revierte funcionalidad.
